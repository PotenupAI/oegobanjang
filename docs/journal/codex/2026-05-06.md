## 오늘 결정한 것

- LangChain에게 routing, blocking, approval final decision 권한을 주지 않는다.
- 기존 `intent_router.py`, `guardrails.py`, `approval_gate.py`, `evidence_logger.py` 흐름은 WorkBridge 안전 계약으로 유지한다.
- LangChain은 나중에 safe tool 호출과 structured judgment output 생성에만 사용한다.
- 실제 LLM/provider 연결은 마지막 단계로 미룬다.
- `009`는 LangChain 없이 provider-agnostic judgment JSON chain으로 닫는다.
- `010`은 judgment JSON을 deterministic risk/report로 변환하는 단계로 닫는다.
- `011`은 `runtime_mode="langchain_judgment"`일 때만 fake judgment/report node가 켜지는 optional runtime 연결로 재정의한다.
- 기존 `011-llm-runtime-integration.md`는 너무 이른 real provider adapter 성격이라 삭제하고 `011-judgment-runtime-mode.md`로 대체한다.
- LangChain adapter는 `012-langchain-judgment-adapter.md`, 실제 provider 연결은 `013-real-llm-provider-integration.md`로 분리한다.

### 5. Mission 011: Judgment Runtime Mode 완료

- `backend/app/schemas/agent.py`의 `AgentRunRequest`에 `runtime_mode`를 추가했다.

```python
runtime_mode: Literal["deterministic", "langchain_judgment"] = "deterministic"
```

- 기본값은 기존 동작 보존을 위해 `deterministic`으로 두었다.
- `backend/app/agent_runtime/graph/workflow.py`에 optional judgment node를 추가했다.
- 기존 workflow 순서는 유지했다.

```txt
mask_payload
-> route_intent
-> guardrail pre-check
-> create_plan
-> execute_plan
-> approval_gate
-> if runtime_mode == "langchain_judgment" and not blocked: judgment node
-> evidence append
-> final_response
-> final guardrail check
```

- `runtime_mode="deterministic"`에서는 judgment/report가 생성되지 않는다.
- `runtime_mode="langchain_judgment"`에서는 fake judgment/report가 생성된다.
- blocked/unsupported request에서는 judgment node가 호출되지 않는다.
- judgment/report 결과를 `final_response`에 포함하도록 했다.
- `llm_judgment_generated`, `risk_flagged` Evidence Log 후보 이벤트를 추가했다.
- `backend/app/agent_runtime/graph/nodes/final_response.py`가 optional `judgment_report`를 포함할 수 있게 수정했다.
- `backend/tests/test_agent_workflow.py`에 runtime mode 테스트를 추가했다.
- Mission 011을 `missions/completed/011-judgment-runtime-mode.md`로 이동했다.
