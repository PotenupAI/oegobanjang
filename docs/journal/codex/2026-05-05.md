## 5/5에 도출한 다음 단계 계획

현재 완료된 “검색 가능한 RAG 기반 + Agent Runtime skeleton” 위에 다음 흐름을 붙이기로 했다.
(검색 기능과 agent 실행 흐름이 있으므로, 그 위에 판단 JSON과 리포트를 얹는 계획)

```txt
사용자 요청
-> intent/router
-> RAG evidence package
-> LLM 판단 JSON 생성
-> parser/guardrail 검증
-> 위험도 분류
-> 기본 리포트 생성
-> approval_required 포함 응답
```

권장 mission 분리:

| Mission | 목적 | 이유 |
|---|---|---|
| `008-rag-evidence-package.md` | 검색 결과를 LLM 입력 Evidence Package로 표준화 | RAG 검색은 되지만 LLM 입력 계약이 없음 (검색 결과를 판단 모델이 읽을 수 있는 JSON으로 고정) |
| `009-llm-judgment-json-chain.md` | prompt, JSON schema, parser, fake LLM test 구성 | 실제 LLM 호출 전에 출력 계약 고정 필요 (LLM이 아무 말이나 하지 않게 구조를 먼저 잠금) |
| `010-risk-report-generation.md` | 판단 JSON을 위험도/기본 리포트로 변환 | 사용자용 산출물 생성 단계 (판단 결과를 담당자가 읽을 리포트로 바꿈) |
| `011-llm-runtime-integration.md` | runtime/provider 연결 | 앞 단계 검증 후 실제 호출 연결 (검증된 계약 뒤에 provider를 붙이는 단계) |

LangChain 도입 원칙:

- 기존 WorkBridge deterministic runtime을 대체하지 않는다. (이미 안전하게 검증된 실행 흐름을 갈아엎지 않음)
- routing, blocking, approval final decision은 기존 WorkBridge runtime이 가진다. (의도 분류, 차단, 승인 판단의 최종권은 기존 코드가 유지)
- LangChain은 RAG 검색, safe tool 사용, structured output judgment report 생성에만 쓴다. (LangChain은 보조 판단/리포트 생성 역할로 제한)
- 실제 provider/API 호출은 마지막 단계로 미룬다. (외부 LLM 호출은 계약과 테스트가 안정된 뒤 연결)
- 테스트에서는 fake model/fake embedding을 사용하고 OpenAI API를 호출하지 않는다. (테스트가 네트워크/API 키에 의존하지 않게 함)
