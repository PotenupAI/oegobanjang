LangChain 1.0 기반 WorkBridge RAG + 판단 체인 실행 계획
Summary
목표는 기존 WorkBridge Agent Runtime을 갈아엎지 않고, 그 위에 LangChain 1.0 structured judgment agent를 추가하는 것이다.
현재 route -> plan -> execute -> approval -> evidence -> final 흐름과 guardrail/approval/evidence 계약은 유지한다.
LangChain은 RAG 검색 -> 안전한 Tool 사용 -> Pydantic structured output -> 판단 리포트 생성에만 사용한다.
Notion에서 확인한 핵심 흐름은 Load -> Split -> Embed -> Store -> Retrieve -> Generate, create_agent, Tool, Structured Outputs, Middleware, Memory, LangGraph다.

비자 가능 확정, 법률/노무 자문, 자동 제출 판단은 schema/prompt 양쪽에서 금지한다.
API/runtime 연결:
AgentRunRequest에 runtime_mode를 추가한다.
허용값: deterministic, langchain_judgment
기본값은 deterministic으로 두어 기존 테스트와 동작을 깨지 않는다.
runtime_mode="langchain_judgment"일 때만 LangChain 판단 agent를 호출한다.
응답에는 judgment_report를 추가하되 기존 final_response, approval, evidence_events는 유지한다.
Evidence/approval:
LangChain agent 호출 전 PII masking을 반드시 수행한다.
unsupported intent는 LangChain 호출 전에 차단한다.
RAG 검색 시 rag_retrieved, LangChain 판단 완료 시 tool_executed 또는 risk_flagged, 최종 응답 시 final_response_generated를 남긴다.
메시지 발송/전문가 전달/export는 여전히 approval_required=true, PENDING이다.

순서는 이렇게 가는 게 좋습니다:
mask_payload
-> route_intent
-> guardrail pre-check
-> create_plan
-> execute_plan
-> approval_gate
-> if runtime_mode == "langchain_judgment" and not blocked:
       run_langchain_judgment
-> evidence append
-> final_response
-> final guardrail check
즉 LangChain은 route 앞이 아니라 execute 뒤, final_response 앞에 들어가는 게 가장 자연스럽습니다.

결론

LangChain 1.0 도입의 1차 목표는 “AI agent로 전체 workflow를 다시 만드는 것”이 아니라, 기존 WorkBridge의 안전한 deterministic runtime 위에 근거 기반 structured judgment report 생성 능력을 얹는 것입니다.

그래서 구현 우선순위는 이 순서가 좋습니다.
